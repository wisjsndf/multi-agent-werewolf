一、 系统定位与交互逻辑
交互时机： 人类玩家（Human-in-the-loop）仅在“轮到自己发言或投票时”可以呼出导师进行对话咨询。

工程考量： 避免随时打断游戏导致的底层异步引擎（asyncio）死锁与状态不一致（薛定谔的局势），确保每次咨询时，上一轮的对局切片（State Commit）已完全闭环。

二、 核心编排框架：LangGraph
架构选型： 放弃单体长 Prompt 和高延迟的纯 ReAct 循环，采用 LangGraph 的状态图（State Graph）进行微服务级别的模块化编排。

全局状态 (Graph State)： 贯穿全图的上下文流，包含人类当前问题、分层记忆数据以及最终的导师输出。

三、 节点职责与意图路由 (Nodes & Routing)
统筹整个团队的是上司 Agent（Boss / Router），它自身不处理繁重的总结或检索任务，而是承担意图识别与任务下发：

意图拦截与护栏 (Guardrails)： 接收人类输入后，通过 Pydantic 强制进行结构化分类。若判定为“闲聊/越界”（如问天气），直接在路由层切断后续高耗能链路，以符合游戏设定的语境进行打断和重定向。

规则查询 (Rule Query)： 若判定为基础常识问题，直接路由至本地规则函数。

局势盘点与推理 (Status & Reasoning)： 若涉及场上局势分析，调取短期记忆与秘书的记账单，执行内部思维链（CoT）推演。

战术求助 (Tactical Advice)： 若询问特定局面打法，路由给检索 Agent (Retriever) 执行 Stage-Aware RAG。

四、 分层记忆系统 (Hierarchical Memory)
抛弃全量文本压缩，采用多轨记忆对抗信息丢失与模型幻觉：

短期记忆 (无损对话)： 完整保留近期原始发言，供上司 Agent 捕捉微表情与逻辑断层。

情景记忆 (骨架账单)： 由秘书 Agent (Secretary) 独立维护。每轮结束时，强制以 JSON 格式冷酷提取结构化关键事件（跳身份、死亡、投票流向）。

语义记忆 (战术库)： 检索 Agent 专属的 FAISS 向量知识库。

工作记忆 (思考草稿)： 上司 Agent 在最终输出前的 CoT 暂存区。

五、 工具箱与物理级防线 (Tool Calling & RBAC)
导师团队通过本地 Python 函数触达游戏客观状态。为了防止大模型产生越权查询底牌的内在幻觉，所有工具必须附带基于角色的访问控制（RBAC）：

get_player_role(player_id): 按需查询自身底牌，降低初始上下文占用。

get_wolf_team_info(player_id): 查询狼队信息。后端硬编码防御逻辑：若调用方非狼人阵营，系统直接阻断请求并返回 Error 抛给大模型。

get_seer_result(player_id): 神职专属情报提取工具，配备同等级的越权拦截。

六、 离线进化机制 (Offline Reflection)
不在实时线上链路中引入 Reflection（反思）节点，以避免引发灾难性的系统延迟，并防止战术建议被“端水化”。

将 Reflection 移至对局结束后的异步离线环境。由专门的复盘 Agent 结合全局上帝视角提取胜负关键点，向量化后入库，实现战术库的自动化迭代。
